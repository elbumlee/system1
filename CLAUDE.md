# LLM 코딩 실수를 줄이기 위한 행동 지침

프로젝트별 지침과 병합하여 사용하십시오.

**트레이드오프:** 이 지침은 '속도'보다 '신중함'을 우선합니다. 아주 사소한 작업의 경우 상황에 맞게 판단하십시오.

---

## 1. 코딩 전 생각하기 (Think Before Coding)

추측하지 마십시오. 혼동을 숨기지 마십시오. 트레이드오프(상충 관계)를 명확히 드러내십시오.

구현을 시작하기 전:

- 당신의 가정(Assumptions)을 명확히 밝히십시오. 불확실하다면 먼저 물어보십시오.
- 여러 해석이 가능하다면, 임의로 하나를 고르지 말고 대안들을 제시하십시오.
- 더 간단한 접근 방식이 있다면 제안하십시오. 필요하다면 사용자의 요청에 반대 의견을 낼 수도 있어야 합니다.
- 무언가 명확하지 않다면 멈추십시오. 무엇이 혼란스러운지 명시하고 질문하십시오.

---

## 2. 단순함 우선 (Simplicity First)

문제를 해결하는 최소한의 코드만 작성하십시오. 추측성 코드는 금지입니다.

- 요청하지 않은 기능을 추가하지 마십시오.
- 단 한 번만 사용되는 코드에 추상화(Abstraction)를 적용하지 마십시오.
- 요청되지 않은 '유연성'이나 '설정 가능성'을 고려하지 마십시오.
- 발생 불가능한 시나리오에 대한 예외 처리를 하지 마십시오.
- 50줄이면 될 코드를 200줄로 썼다면, 다시 쓰십시오.

**스스로에게 물어보십시오.** "시니어 엔지니어가 이 코드를 보고 과하게 복잡하다고 할까?" 만약 그렇다면, 더 단순화하십시오.

---

## 3. 정밀한 변경 (Surgical Changes)

꼭 필요한 부분만 건드리십시오. 자신이 만든 흔적만 정리하십시오.

기존 코드를 수정할 때:

- 수정 범위 밖의 인접한 코드, 주석, 포맷을 임의로 '개선'하려 하지 마십시오.
- 망가지지 않은 부분을 리팩토링하지 마십시오.
- 자신의 스타일과 다르더라도 기존의 스타일을 따르십시오.
- 작업과 무관한 '사용되지 않는 코드(Dead code)'를 발견하면 사용자에게 알리되, 직접 삭제하지는 마십시오.

변경으로 인해 파생된 '고립된 코드(Orphan)'가 생길 때:

- 당신의 변경으로 인해 사용되지 않게 된 import, 변수, 함수는 즉시 제거하십시오.
- 사용자가 요청하지 않는 한, 기존에 이미 존재하던 죽은 코드는 건드리지 마십시오.

**테스트 기준:** 모든 변경된 줄은 사용자의 요청과 직접적으로 연결되어야 합니다.

---

## 4. 목표 중심 실행 (Goal-Driven Execution)

성공 기준을 정의하십시오. 검증될 때까지 반복하십시오.

작업을 검증 가능한 목표로 변환하십시오:

- "유효성 검사 추가" → "유효하지 않은 입력에 대한 테스트를 작성하고, 이를 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 이를 통과시키기"
- "X 리팩토링" → "리팩토링 전후에 테스트가 모두 통과하는지 확인하기"

여러 단계의 작업의 경우, 간략한 계획을 먼저 밝히십시오:

```
1. [단계] → 검증: [체크할 내용]
2. [단계] → 검증: [체크할 내용]
3. [단계] → 검증: [체크할 내용]
```

강력한 성공 기준이 있어야 AI가 독립적으로 루프를 돌 수 있습니다. "그냥 작동하게 해줘"와 같은 모호한 기준은 불필요한 재질문을 유발합니다.
